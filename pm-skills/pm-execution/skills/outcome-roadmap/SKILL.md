# Outcome Roadmap

Use when: the user wants to create a roadmap, convert a feature list into an outcome-based plan, or communicate product direction to stakeholders.

Trigger: "roadmap", "product roadmap", "outcome roadmap", "quarterly roadmap", "roadmap planning", "prioritize features"

## Framework

### Feature Roadmap vs. Outcome Roadmap

| Feature Roadmap | Outcome Roadmap |
|-----------------|-----------------|
| "Build feature X" | "Achieve outcome Y" |
| Commits to specific solutions | Commits to solving problems |
| Gets outdated quickly | Stays valid even as solutions change |
| Creates stakeholder expectation debt | Enables team autonomy |
| Focuses team on output | Focuses team on impact |

### Outcome Roadmap Structure

**Time Horizons:**
- **Now** (current quarter): High confidence, committed work
- **Next** (next 1–2 quarters): Medium confidence, directional
- **Later** (3–6 months+): Low confidence, aspirational themes

**For each time horizon, define:**
1. **Outcome**: The metric or behavior change we're targeting
2. **Opportunity**: The customer need we're addressing
3. **Bets**: The solution approaches we'll explore (not features — hypotheses)
4. **OKR Link**: Which company or team OKR this contributes to

### Stakeholder Communication Tips

When presenting the outcome roadmap:
1. Lead with the business goal, not the features
2. Explain the "why" before the "what"
3. Show OKR alignment explicitly
4. Frame Later items as bets, not promises
5. Always present what's NOT on the roadmap and why (creates trust)

## Output Format

Generate the roadmap in three-horizon format (Now / Next / Later). For each item include: outcome, opportunity, bets, OKR link, and confidence level. Include a one-page executive summary version at the top (3–5 bullet points max). End with "What's NOT on this roadmap" section listing 3–5 deliberate exclusions.
